# InboxDock 产品化与灵活 Vault 适配设计

日期：2026-07-23

状态：已与产品所有者确认，等待实施计划

目标版本：v0.3.0 公开测试版，后续按 v0.4.0、v0.5.0、v1.0.0 演进

## 1. 背景

InboxDock 当前已经是一个可独立运行的 Windows 桌面收集工具：Obsidian 关闭时仍可接收文字、链接、剪贴板图片和文件；材料先进入本地暂存区，用户确认后再写入 Vault；窗口支持四边吸附、自动收边、托盘运行和跨重启暂存。

现有实现主要按照产品所有者自己的 Vault 结构设计，默认依赖以下概念和路径：

- `00 Inbox收件箱`
- `01 Daily日常`
- `10 Knowledge Hub/Templates/Daily.md`
- `05 Resources/Attachments`

这套结构适合当前个人工作流，但不适合直接要求其他 Obsidian 用户采用。Obsidian 用户可能使用 PARA、Johnny Decimal、Zettelkasten、Daily-only、纯标签、扁平目录或自定义项目结构。产品化改造必须保留当前用户体验，同时消除对 Inbox、Daily、Project 等特定知识管理方法的硬编码。

当前公开发布链路也未完整反映主分支能力。产品化版本需要同时补齐安装、升级、首次配置、诊断、隐私说明、版本发布和真实用户验证。

## 2. 产品定位

InboxDock 的定位是：

> 一个独立于 Obsidian 运行的 Windows 桌面材料收件箱。它在不打断当前工作的情况下接住文字、链接、截图和文件，并由用户决定何时、以何种格式写入 Vault。

首要目标用户是 Windows 上的重度 Obsidian 用户。他们重视：

- 系统级快速呼出，不必切换到 Obsidian。
- Obsidian 关闭时仍可收集。
- 本地优先、无账号、无默认遥测。
- 不丢材料，不擅自移动或删除原文件。
- 能适配已有 Vault，而不是要求重建目录结构。
- 收集动作足够快，但复杂配置只在需要时出现。

InboxDock 不定位为完整 Markdown 编辑器、Vault 浏览器、知识整理软件或 Obsidian 替代品。

## 3. 产品原则

后续功能需优先满足以下原则：

1. **先接住，再整理**：用户可以只暂存，不必在灵感出现时立即决定最终目录。
2. **适配用户的 Vault**：Inbox、Daily、Project 只是可选示例，不是产品内置方法论。
3. **数据安全优先**：写入失败时保留暂存材料，任何目标路径不得离开 Vault。
4. **默认简单，高级可选**：首次使用只需选择 Vault 和默认保存位置。
5. **系统级低打扰**：贴边、快捷键、自启动和状态提示服务于快速收集，不制造持续干扰。
6. **本地优先**：默认不上传笔记、不监控剪贴板、不收集使用内容。
7. **克制扩展**：不因少数高级场景引入脚本平台、规则市场或完整知识管理功能。

## 4. 方案选择

### 4.1 未选择：固定 Vault 预设

只提供 Inbox、PARA、Daily-only 等固定预设，上手简单，但无法覆盖命名、目录、附件和写入方式的广泛差异。预设可作为示例，不能作为底层模型。

### 4.2 未选择：完全自由的规则系统

允许用户编写任意脚本、正则和自动路由规则，能力强但配置成本高，容易让 InboxDock 变成面向程序员的自动化平台。

### 4.3 采用：简单默认值 + 自定义收集目标 + 可选高级模板

产品底层使用通用“收集目标”，首次配置只暴露必要选项；高级用户可按需设置模板、路径和附件策略。该方案兼顾上手速度、Vault 兼容性和长期扩展。

## 5. v0.3.0 范围

v0.3.0 目标是把个人工具升级为可信任的公开测试版，包含：

- 通用收集目标。
- 现有设置和暂存数据的安全迁移。
- 简化首次配置。
- 主界面目标选择。
- 可配置的 5 秒、10 秒、30 秒或永不自动收回。
- 图钉固定展开。
- 用户自定义全局呼出快捷键。
- 用户可控制的开机自启动。
- 单实例运行。
- 贴边把手显示待处理数量和失败状态。
- 基本批量处理。
- 保存成功、失败和撤销反馈。
- 版本、日志、诊断和更新检查。
- Windows 安装版与便携版。
- 标准版本标签、自动发布、README 和发布说明。

v0.3.0 不包含 OCR、语音、AI 自动分类、网页摘要、桌面宠物、多 Vault 同时工作和脚本扩展。

## 6. 配置模型

### 6.1 VaultProfile

v0.3.0 主界面只操作一个当前 Vault，但底层使用 `VaultProfile`，为后续多 Vault 切换保留清晰边界。

建议字段：

- `Id`
- `Name`
- `VaultPath`
- `DefaultTargetId`
- `DefaultAttachmentPolicy`
- `CaptureTargets`
- `Theme`
- `AlwaysOnTop`
- `LaunchAtSignIn`
- `AutoHideDelay`
- `GlobalHotkey`
- `WindowState`

v0.3.0 不提供多个 Vault 同时监听或自动分发。

### 6.2 CaptureTarget

收集目标替代固定 Inbox、Daily 和 Project 概念。

建议字段：

- `Id`
- `Name`
- `Icon`
- `Color`
- `WriteMode`
- `PathTemplate`
- `FileNameTemplate`
- `ContentTemplate`
- `AttachmentPolicy`
- `InsertionMode`
- `HeadingName`
- `PostCaptureBehavior`
- `IsDefault`
- `SortOrder`
- `Revision`

`Revision` 用于识别目标是否在上次预览后发生过修改。新目标或修改过的目标首次使用时强制显示写入预览。

### 6.3 WriteMode

v0.3.0 只支持四种写入方式：

1. `AppendToFile`：追加到固定 Markdown 文件。
2. `AppendToPeriodicFile`：按照日期路径追加到周期笔记。
3. `CreateNote`：每次创建一篇新笔记。
4. `StagingOnly`：只保留在 InboxDock 暂存区。

Inbox 和 Daily 可以作为首次配置生成的目标名称，但不作为写入类型。

### 6.4 AttachmentPolicy

支持：

- `FollowObsidian`：沿用可安全读取到的 Obsidian 附件设置。
- `FixedDirectory`：固定 Vault 相对目录。
- `DatedDirectory`：带日期变量的 Vault 相对目录。
- `BesideNote`：放在目标笔记旁的指定子目录。
- `StagingOnly`：附件暂不进入 Vault。

任何渲染后的路径必须：

- 为非空 Vault 相对路径。
- 解析后仍位于 Vault 根目录内。
- 不覆盖已有文件。
- 对非法字符、保留名称和超长路径给出可理解错误。

### 6.5 模板变量

v0.3.0 只提供有限变量，不支持脚本执行：

- `{{content}}`
- `{{title}}`
- `{{url}}`
- `{{note}}`
- `{{files}}`
- `{{date}}`
- `{{time}}`
- `{{timestamp}}`
- `{{source}}`
- `{{target}}`

日期和时间允许简单 .NET 格式，例如 `{{date:yyyy-MM-dd}}`、`{{time:HH:mm}}`。未知变量、非法格式或缺少必要值时停止写入并保留材料。

高级模板在设置中默认折叠。v0.3.0 不提供 JavaScript、正则路由或条件模板。

## 7. 统一材料模型与数据流

文字、链接、图片和文件先归一化为通用材料，再由目标决定最终写入方式。

建议材料字段：

- `Id`
- `Kind`
- `Title`
- `Content`
- `Url`
- `Note`
- `StagedFiles`
- `CreatedAt`
- `PreferredTargetId`
- `Status`
- `LastError`

完整流程：

1. 用户输入、粘贴、拖放或通过快捷键呼出输入。
2. InboxDock 识别材料类型。
3. 文件和图片完整复制到本地暂存区后才创建卡片。
4. 卡片记录可选的首选目标，但不提前计算最终路径。
5. 用户确认时使用当前目标生成笔记、附件路径和 Markdown 预览。
6. 系统校验 Vault、路径、重名、模板和写入位置。
7. 使用可恢复、原子化的方式写入附件和 Markdown。
8. 成功后显示目标、撤销和“在 Obsidian 中打开”。
9. 成功完成且不再需要撤销后，清理 InboxDock 暂存副本。
10. 任一步失败时保留完整卡片、备注和暂存文件。

待处理材料只保存 `PreferredTargetId`，不保存已经解析的绝对写入路径。若目标被删除，要求用户重新选择；材料本身不得丢失。

## 8. 首次配置

首次启动只包含两步。

### 8.1 选择 Vault

验证：

- 目录存在。
- 包含 `.obsidian`。
- 当前用户具备必要写入权限。

### 8.2 选择默认保存方式

提供：

- 选择现有 Markdown 文件并追加。
- 选择文件夹并为每次收集创建新笔记。
- 暂时只使用 InboxDock 暂存区。

如果能读取到 Obsidian Daily Notes 配置，则额外建议创建“今日日记”目标。自动发现只用于建议，必须经过用户确认，InboxDock 不自动修改 Vault 设置。

首次配置不要求用户设置模板、Frontmatter、附件变量或快捷键。

## 9. 主界面与目标管理

### 9.1 主界面

保留当前材料列表和输入体验，在输入区附近增加：

```text
保存到：收件箱 ▼
```

下拉菜单显示最近使用的目标、全部目标和“管理目标”。正常收集仍保持：输入或粘贴后按 `Ctrl+Enter`，保存成功后按照窗口设置收回。

### 9.2 目标管理

目标列表只提供：

- 新建
- 编辑
- 删除
- 设为默认
- 调整顺序

新建时先选择追加到文件、追加到日记、创建新笔记或只暂存，然后填写名称和目标位置。模板、附件策略和标题插入规则位于折叠的高级区域。

### 9.3 写入预览

正常使用不增加每次确认步骤。以下情况强制显示预览：

- 第一次使用新目标。
- 目标配置或模板发生过修改。
- 目标路径不存在或不再可写。
- 文件名或附件重名。
- 模板渲染失败。
- 指定标题不存在。

预览显示最终笔记路径、附件目录和生成的 Markdown，不执行写入。

## 10. 窗口行为、快捷键与开机自启动

### 10.1 自动收回

设置选项：

- 5 秒，默认值。
- 10 秒。
- 30 秒。
- 永不自动收回。

输入、输入法组合、拖放、材料复制、确认面板、设置、错误处理和鼠标仍在窗口内时暂停倒计时。暂停结束后重新开始完整倒计时。用户点击图钉后保持展开，直到取消固定。

保存成功后可在约 1.5 秒反馈完成后收回；鼠标进入反馈区域或用户点击撤销时暂停收回。保存失败时禁止自动收回。

### 10.2 全局快捷键

默认建议为 `Ctrl+Shift+Space`，用户可修改或关闭。

设置界面通过实际按键捕获组合键，不要求用户输入字符串。保存时尝试注册；若注册失败或冲突：

- 保留此前仍有效的快捷键。
- 不保存无效组合。
- 告知用户该组合可能被系统或其他应用占用。

快捷键行为：

- 收边时展开并聚焦输入框。
- 已展开时聚焦输入框。
- 被其他窗口遮挡时恢复到前台。

v0.3.0 只提供一个系统级呼出快捷键。

### 10.3 单实例

第二次启动 InboxDock 时，不创建另一个托盘图标、计时器或暂存服务，而是通知现有实例展开并聚焦。

### 10.4 开机自启动

设置页面提供明确开关：

```text
登录 Windows 后自动启动 InboxDock
```

规则：

- 默认关闭。
- 用户开启后注册当前可执行文件的登录启动项。
- 用户关闭后彻底移除 InboxDock 创建的启动项。
- 注册或移除失败时恢复开关原状态并显示错误。
- 安装版卸载时移除启动项。
- 便携版移动或删除后，下次运行要识别旧启动路径已失效，并允许修复。
- 自启动后按用户最近的窗口状态运行；首版默认可显示主窗口，后续根据反馈再考虑静默启动设置。

自启动设置不得与安装程序选项形成两个互相覆盖、状态不一致的来源。应用设置是运行时状态的唯一展示入口。

## 11. 状态挂件与待处理体验

v0.3.0 不开发独立桌面宠物，而是升级现有贴边把手：

- 无待处理材料：普通箭头。
- 有待处理材料：显示数量。
- 有写入失败：显示红点或错误状态。
- 正在接收文件：显示短暂接收状态。

待处理列表补充材料类型、摘要、收集时间、首选目标、状态和失败原因。顶部显示待处理数量和最早材料时间。

基本批量操作：

- 将选中或全部兼容材料收进当前目标。
- 删除选中材料的 InboxDock 暂存副本。

批量写入前必须确认目标支持相应材料类型。删除不影响原始文件，且保持现有二次确认。

## 12. 成功、失败与撤销

### 12.1 成功

成功后显示：

```text
已保存到“收件箱”
撤销｜在 Obsidian 中打开
```

“在 Obsidian 中打开”使用最终目标笔记路径。“撤销”只撤销 InboxDock 刚完成且能安全识别的写入，不删除写入前已经存在的用户内容。

### 12.2 失败

失败时使用用户可理解的说明：

```text
没有写入成功，材料仍安全保存在 InboxDock。
```

提供重试、更换目标和复制错误信息。普通界面不直接展示完整异常堆栈。

## 13. 设置页面

设置只分为四部分：

1. Vault
2. 收集目标
3. 窗口与快捷键
4. 关于与诊断

“窗口与快捷键”包括始终置顶、自动收回时间、图钉行为、全局快捷键和开机自启动。

“关于与诊断”包括版本号、检查更新、项目地址、打开日志目录、打开暂存目录、复制诊断信息、隐私说明和许可证。

## 14. 配置迁移

新配置包含明确的 `SchemaVersion`。首次读取旧设置时：

1. 先备份旧 `settings.json`。
2. 将旧 Vault、Inbox、Daily、模板、附件、窗口和自启动设置映射到 `VaultProfile`。
3. 创建“收件箱”和“今日日记”两个目标。
4. 保留现有暂存卡片和文件。
5. 原子写入新配置。
6. 重新读取并验证新配置。
7. 失败时恢复旧配置并停止破坏性操作。

迁移不得删除旧备份。旧版暂存数据缺少新字段时使用安全默认值。

## 15. 安装、更新与发布

### 15.1 发布文件

正式版本提供：

- `InboxDock-Setup-0.3.0.exe`
- `InboxDock-Portable-0.3.0.zip`
- `SHA256SUMS.txt`

安装版提供开始菜单入口、可选桌面快捷方式、正常卸载和自启动清理。卸载默认不删除用户设置、日志和暂存材料；删除用户数据必须由用户明确选择。

### 15.2 版本与标签

使用语义化版本号和 `v0.3.0` 形式的 Git 标签。修复版本递增补丁号，新增兼容功能递增次版本号，达到稳定标准后发布 v1.0.0。

### 15.3 更新检查

v0.3.0 只检查 GitHub 最新 Release：

- 最多每天自动检查一次。
- 可在设置中关闭。
- 有新版本时展示安静提示并打开下载页面。
- 不自动下载或安装。
- 无网络或 GitHub 不可用时不阻塞启动和收集。

### 15.4 README 与 Release

仓库名称建议统一为 InboxDock，描述清楚 Windows、Obsidian、独立运行和本地优先。README 第一屏包含定位、截图、下载入口、30 秒上手、隐私承诺和已知限制。

每次 Release 说明必须列出新功能、改进、修复、升级提醒和下载选择，不再使用模糊说明。

## 16. 隐私与诊断

默认行为：

- 无账号。
- 无遥测。
- 无广告。
- 不上传笔记和附件。
- 不后台监控剪贴板。
- 不扫描与当前操作无关的 Vault 内容。
- 网络仅用于用户允许的更新检查。

日志不得记录笔记正文、附件内容、剪贴板内容和用户输入。诊断信息可包含版本、Windows 版本、架构、Vault 是否存在和可写、暂存数量及最近错误类型；私人路径需遮蔽。

## 17. 代码边界

继续保持 `InboxDock.Core` 与 WPF UI 分离。建议增加或演进以下组件：

- `VaultDiscovery`：只读发现可安全使用的 Obsidian 配置。
- `VaultProfileStore`：保存 Vault 配置和 SchemaVersion。
- `CaptureTargetStore`：管理收集目标。
- `TemplateRenderer`：渲染有限模板变量。
- `TargetPathResolver`：计算并验证笔记和附件路径。
- `CapturePreviewService`：生成无副作用预览。
- `TargetWriteService`：执行追加、新建和附件写入。
- `SettingsMigration`：迁移旧配置。
- `WriteUndoService`：记录可安全撤销的最近写入。
- `GlobalHotkeyService`：注册、更新和释放系统快捷键。
- `LaunchAtSignInService`：注册、校验和移除开机自启动。
- `SingleInstanceService`：复用现有进程。
- `UpdateCheckService`：执行可关闭、非阻塞的版本检查。

纯路径、模板、迁移、状态和写入决策尽量放在 Core 中并通过单元测试验证。WPF 负责选择、展示、确认和系统窗口交互。

## 18. 错误处理要求

必须覆盖：

- Vault 被移动、删除或失去写权限。
- Vault 位于同步目录且文件临时占用。
- 目标或模板被用户删除。
- 指定标题不存在。
- 附件或笔记重名。
- 磁盘空间不足。
- 暂存文件缺失或损坏。
- 快捷键注册冲突。
- 开机自启动注册失败或路径失效。
- 更新检查断网、超时或 GitHub 不可用。
- 写入过程中应用异常退出。

所有错误的共同规则是：不覆盖现有用户内容，不删除未确认的暂存材料，不因网络问题阻止本地收集。
